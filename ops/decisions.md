# WCRC 결정 기록

이 파일은 코드가 바뀌어도 유지되어야 하는 설계 결정을 기록한다.
결정 이력의 원본은 Git이고, Notion은 요약과 회고용 보조 문서로만 둔다.

## 2026-04-30 - Git 추적 운영 기록 원본

- 상태: 채택
- 결정: WCRC 운영 기록은 `Motor/ops/`에 두고, Public&Private private console에서 읽기 전용으로 보여준다.
- 이유: Git은 정확한 변경 이력과 롤백 맥락을 남길 수 있고, 콘솔은 현장에서 빠르게 확인하는 화면으로 적합하다.
- 다시 볼 조건: 현장 기록 속도가 느려지거나, 대시보드에서 직접 기록해야 할 필요가 생길 때.

## 2026-04-30 - 적재함 칸 번호 체계

- 상태: 제안
- 결정: 고정된 적재함을 가로 4칸, 세로 2칸으로 모델링한다.
- 기준:

```text
camera side

1  2  3  4
5  6  7  8
```

- 이유: signature만 보고 집는 방식은 빠르지만, 같은 색이 반복되거나 Pixy가 잠깐 오인식하면 어떤 물리 칸에서 가져왔는지 증명하기 어렵다.
- 구현 규칙: 먼저 `slot -> signature` Serial 로그를 추가하고, 반복 현장 테스트에서 안정성이 확인된 뒤 집기 로직에 연결한다.
- 다시 볼 조건: 현장 테스트 3회 연속으로 칸 추정이 틀릴 때.

## 2026-04-30 - 미션 수행존 배치는 EEPROM 자세로 관리

- 상태: 채택
- 결정: 미션 수행존 각 칸 배치는 로봇 본체를 칸마다 움직이는 방식보다 EEPROM 자세 7~14로 관리한다.
- 이유: 본체 이동은 바퀴 미끄러짐이 누적된다. 팔 자세는 칸별 튜닝과 롤백이 더 쉽다.
- 다시 볼 조건: 특정 칸이 매니퓰레이터 자세만으로 안전하게 닿지 않을 때.

## 2026-04-30 - Pixy signature 제약

- 상태: 채택
- 결정: Pixy `sigmap`은 이미 학습된 signature를 필터링하는 용도로만 보고, 런타임 색상 프로필 교체 수단으로 보지 않는다.
- 이유: PixyMon에서 학습 가능한 signature 수에 제약이 있고, 코드 필터만으로 독립적인 두 번째 인식 세트가 생기지는 않는다.
- 다시 볼 조건: 경기에서 학습 가능한 signature 수보다 더 많은 독립 클래스가 필요할 때.
